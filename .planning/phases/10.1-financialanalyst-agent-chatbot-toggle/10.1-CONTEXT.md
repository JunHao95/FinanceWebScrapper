# Phase 10.1: FinancialAnalyst Agent & Chatbot Toggle - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a FinancialAnalyst agent alongside the existing QuantAssistant in the chatbot widget. Users switch between agents via a toggle in the widget header. Each agent has a distinct persona, system prompt, and independent chat history. New capabilities (e.g., additional agents, persistent cross-session history) are out of scope.

</domain>

<decisions>
## Implementation Decisions

### Toggle control design
- Pill-shaped tabs in the chat header row: `[QuantAssistant] [FinancialAnalyst]` side by side with the close button
- Active pill is highlighted; inactive pill is dimmed
- Header `h3` title updates dynamically to the active agent name when switching
- Subtitle div also updates per agent to reflect the active agent's tagline

### History handling on agent switch
- Separate histories per agent — each agent maintains its own independent message thread in JS memory
- Switching agents swaps in that agent's message list; switching back restores the previous thread
- Histories survive widget open/close within the same page session (stored in JS variables, not DOM)
- No dividers or merged views — clean separation between the two conversation threads

### FinancialAnalyst persona
- **Focus**: Company fundamentals (P/E, revenue, earnings), sector analysis, macro trends, investment thesis — the qualitative/fundamental side
- **Contrast with QuantAssistant**: QuantAssistant stays on quant math, stochastic models, derivatives; FinancialAnalyst covers the fundamental/macro domain
- **Communication style**: Concise bullet-point analyst style — short, structured responses like a sell-side analyst's note (key metrics, quick summary, actionable insight)
- **Header name**: 📈 FinancialAnalyst
- **Header subtitle**: Company fundamentals & market analysis
- **Greeting message**: Short intro matching the analyst style

### Backend routing
- Single `/api/chat` endpoint; frontend sends `{ message, agent: 'quant' | 'financial' }` in the request body
- Backend selects `system_prompt` based on the `agent` field — no new routes needed
- Frontend sends last N messages as context for multi-turn coherence (JS per-agent message store doubles as context array)
- `agent` field defaults to `'quant'` if absent (backward compatible with existing behavior)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `static/js/chatbot.js`: All widget HTML is generated in JS via `chatContainer.innerHTML`. Toggle pills can be added directly into that template string in the header div.
- `appendMessage(sender, text)`: Existing helper appends a message div to `#chatbot-messages`. Swapping agent history = replacing `messagesContainer.innerHTML` or maintaining two DOM fragment arrays.
- `appendTypingIndicator()` / `removeMessage()`: Reusable as-is for both agents.
- `renderMarkdown(text)`: Shared renderer works for both agents.

### Established Patterns
- All widget state is in-memory JS variables (no localStorage, no server state). Per-agent history arrays follow the same pattern.
- Chat window toggle: `chatWindow.classList.toggle('chatbot-hidden')` — existing open/close pattern unchanged.
- Fetch pattern: `fetch('/api/chat', { method: 'POST', body: JSON.stringify({ message: text }) })` — extend to include `agent` field.

### Integration Points
- `webapp.py` `/api/chat` route at line ~1963: reads `data.get("message")` and `system_prompt` variable. Add `agent = data.get("agent", "quant")` and a prompt lookup dict keyed by agent name.
- CSS: chatbot styles in `static/css/styles.css` — add pill tab styles there (active/inactive states).
- `#chatbot-header` div in the injected HTML: receives the pill tabs + updated title/subtitle logic.

</code_context>
