## Summary

Phase 10.1 extends the existing chatbot widget — a self-contained JS module in `static/js/chatbot.js` — with a second agent persona and a toggle UI. The current widget already handles all widget HTML through `chatContainer.innerHTML`, making it straightforward to inject two pill-tab buttons into the header div without touching any HTML templates. All widget state is stored in JS closures (no localStorage, no server session), so per-agent history arrays follow identically the same in-memory pattern already in use.

The Flask `/api/chat` route at line 1963 of `webapp.py` already contains the full LLM dispatch logic (Groq cloud path and Ollama local path). The only backend change is to read an `agent` field from the request body and look up the correct `system_prompt` from a dict — a single-branch addition that is fully backward-compatible because the field defaults to `'quant'` when absent.

The existing dark-theme CSS in `styles.css` already uses `var(--primary-color)` and `var(--border-glass)` tokens, making it simple to add active/inactive pill styles that naturally match the widget aesthetic. No third-party libraries, build steps, or new routes are required.

**Primary recommendation:** Implement all changes in three discrete, independently-testable steps — (1) CSS pill styles, (2) JS agent state + header swap + history swap, (3) backend `agent` field routing — in that order.

### Pattern 1: Per-Agent Message Store (JS in-memory)

**What:** Two parallel arrays — one per agent — that shadow `messagesContainer` children. On agent switch, serialize the current DOM children into the departing agent's array, then replay the arriving agent's array into the container.

**When to use:** Any time multiple conversation threads must coexist in a single widget instance without server state.

**Example:**
```javascript
// Source: derived from existing chatbot.js appendMessage pattern

const agentHistories = {
    quant: [],      // each entry: { sender: 'user'|'bot', text: string }
    financial: []
};
let activeAgent = 'quant';

function switchAgent(newAgent) {
    if (newAgent === activeAgent) return;

    // Serialize current thread into departing agent's store
    agentHistories[activeAgent] = captureMessages();

    activeAgent = newAgent;

    // Replay arriving agent's stored messages
    messagesContainer.innerHTML = '';
    agentHistories[activeAgent].forEach(({ sender, text }) => appendMessage(sender, text));

    // Update header
    updateHeader(newAgent);
}
```

**Note on `captureMessages()`:** Because `appendMessage` builds DOM nodes, the simplest capture is to save `{ sender, text }` objects at write-time (when calling `appendMessage`) rather than scraping the DOM at switch-time. This avoids HTML round-tripping and preserves the original unrendered text for future context arrays.

### Pattern 2: Context Array for Multi-Turn Coherence

**What:** The per-agent history array doubles as the LLM context payload — last N entries are sent as `messages` alongside the new user message.

**When to use:** Whenever multi-turn coherence is needed without a server session store.

**Example:**
```javascript
// Source: extension of existing fetch pattern in chatbot.js line 91-95

const MAX_CONTEXT = 10; // last 10 turns

const contextMessages = agentHistories[activeAgent]
    .slice(-MAX_CONTEXT)
    .map(({ sender, text }) => ({
        role: sender === 'user' ? 'user' : 'assistant',
        content: text
    }));

body: JSON.stringify({
    message: text,
    agent: activeAgent,        // 'quant' | 'financial'
    history: contextMessages   // optional multi-turn context
})
```

### Pattern 3: Backend System Prompt Dispatch

**What:** A dict keyed by agent name holds the full system prompt for each persona. The route reads `agent` from the request body and looks up the prompt — no conditional branching in the LLM call itself.

**When to use:** Any time a single endpoint must serve multiple personas without route proliferation.

**Example:**

### Pattern 5: Greeting Initialization on First Switch

**What:** Each agent's history array starts empty. On first load, QuantAssistant's history is pre-seeded with its greeting. On first switch to FinancialAnalyst, its history is also empty, so a greeting is injected before replaying. This avoids a blank chat area.

**When to use:** Any time an agent is visited for the first time in a session.

**Example:**
```javascript
const AGENT_GREETINGS = {
    quant: "Hello! I'm QuantAssistant. How can I help you with your financial analysis today?",
    financial: "Hi, I'm FinancialAnalyst. Ask me about company fundamentals, sector trends, or macro outlook."
};

// On switch: if target history is empty, seed greeting
if (agentHistories[newAgent].length === 0) {
    appendMessage('bot', AGENT_GREETINGS[newAgent]);
}
```

### Anti-Patterns to Avoid

- **Scraping DOM to capture history on switch:** The DOM contains rendered HTML (markdown → HTML), not the original text. Capturing at write-time is safer and preserves clean text for context arrays.
- **Adding a new Flask route for FinancialAnalyst:** A second route (`/api/chat_financial`) duplicates LLM dispatch logic. The `agent` field pattern keeps the endpoint surface area minimal.
- **Storing history in `localStorage`:** The CONTEXT.md explicitly scopes history to JS memory for this phase. localStorage introduces persistence concerns (stale cross-session data, quota limits) that are out of scope.
- **Using `messagesContainer.innerHTML = ''` to capture departing agent messages:** This destroys the DOM before capturing. Always save `{ sender, text }` at `appendMessage` call time.

## Common Pitfalls

### Pitfall 1: History Capture Timing

**What goes wrong:** If history is captured by reading `messagesContainer.innerHTML` on switch, the bot-message HTML contains rendered markdown (`<strong>`, `<ul>`, etc.) rather than original text. When re-injecting, `appendMessage` runs `renderMarkdown` again on HTML — producing double-escaped or broken output.

**Why it happens:** `appendMessage` uses `msgDiv.innerHTML = renderMarkdown(text)` for bot messages, so the DOM holds HTML not raw text.

**How to avoid:** Push `{ sender, text }` objects into the history array inside `appendMessage` itself (write-time capture), before any rendering. Replay by calling `appendMessage(sender, text)` normally.

**Warning signs:** Bot messages showing raw `<strong>` or `&lt;ul&gt;` tags after agent switch.

### Pitfall 2: `activeAgent` Used Before Pill Event Binds

**What goes wrong:** If `activeAgent` is referenced in `sendMessage` before the pill click handlers are wired up, the variable may be `undefined` and the `agent` field in the fetch body will be `undefined` — causing the backend to pick the default but hiding the bug silently.

**Why it happens:** `let activeAgent = 'quant'` must be declared in the same closure scope that `sendMessage` reads from, before `sendMessage` is defined.

**How to avoid:** Declare `activeAgent` and `agentHistories` at the top of the `DOMContentLoaded` callback, before all function definitions.

### Pitfall 3: Header IDs Must Be Present After `innerHTML` Set

**What goes wrong:** `updateHeader()` calls `document.getElementById('chatbot-agent-name')` — but if the template string in `chatContainer.innerHTML` assigns an `id` that was not given to the h3 element, `getElementById` returns `null` and the update silently fails.

**Why it happens:** The current `chatbot.js` h3 has no `id`. The new template must add `id="chatbot-agent-name"` and `id="chatbot-agent-subtitle"` explicitly.

**How to avoid:** Add the IDs in the template string, then look them up by ID in `updateHeader()`. Verify IDs match exactly (case-sensitive).

**Warning signs:** Header still shows "QuantAssistant" after switching to FinancialAnalyst.

### Pitfall 4: Close/Re-open Clears Active Agent State

**What goes wrong:** If `chatWindow.classList.toggle('chatbot-hidden')` is the only toggle mechanism and no DOM is destroyed, histories survive correctly. But if the widget is ever re-created (e.g., a page navigation that reinjects the container), all JS state is lost.

**Why it happens:** Would only matter if `DOMContentLoaded` fires again. In the current SPA-like Flask app with full page reloads per navigation, this is not an issue.

**How to avoid:** Confirm that widget open/close only toggles `chatbot-hidden` (it does — line 43-47 of chatbot.js). No state reset risk.

### Pitfall 5: Pill Tab Overflow in Narrow Widget Header

**What goes wrong:** Adding two pill buttons beside the existing close button in a 420px-wide header can crowd the layout, especially on narrower viewports where `max-width: calc(100vw - 40px)` applies.

**Why it happens:** The current header uses `justify-content: space-between` with a single left-side block and a single right-side close button. Adding a pill-tab group in the middle requires a three-column layout.

**How to avoid:** Change the header to a three-slot flex layout: `[title-info] [agent-tabs (centered)] [close-btn]`. Use `flex: 1` on the title block to push tabs to center, or set a fixed pill group width. Keep pill labels short (no emojis inside the pills — the `📈` emoji lives in the h3 title only).

### JS Agent State and Switch Handler

```javascript
// Source: chatbot.js — add after element lookups, before sendMessage definition

const AGENT_CONFIG = {
    quant: {
        name: '🤖 QuantAssistant',
        subtitle: 'AI-powered financial analysis',
        greeting: "Hello! I'm QuantAssistant. How can I help you with your financial analysis today?"
    },
    financial: {
        name: '📈 FinancialAnalyst',
        subtitle: 'Company fundamentals & market analysis',
        greeting: "Hi, I'm FinancialAnalyst. Ask me about company fundamentals, sector trends, or macro outlook."
    }
};

const agentHistories = { quant: [], financial: [] };
let activeAgent = 'quant';

// Seed QuantAssistant greeting into its history (mirrors existing static greeting div)
agentHistories.quant.push({ sender: 'bot', text: AGENT_CONFIG.quant.greeting });

function updateHeader(agent) {
    document.getElementById('chatbot-agent-name').textContent = AGENT_CONFIG[agent].name;
    document.getElementById('chatbot-agent-subtitle').textContent = AGENT_CONFIG[agent].subtitle;
    document.querySelectorAll('.agent-pill').forEach(pill => {
        pill.classList.toggle('active', pill.dataset.agent === agent);
    });
}

function switchAgent(newAgent) {
    if (newAgent === activeAgent) return;
    // Capture current messages at switch time is not needed —
    // agentHistories[activeAgent] is kept up-to-date by appendMessage
    activeAgent = newAgent;
    messagesContainer.innerHTML = '';
    if (agentHistories[activeAgent].length === 0) {
        appendMessage('bot', AGENT_CONFIG[activeAgent].greeting);
    } else {
        agentHistories[activeAgent].forEach(({ sender, text }) => appendMessage(sender, text, true));
    }
    updateHeader(activeAgent);
}

// Wire pill clicks
document.getElementById('chatbot-agent-tabs').addEventListener('click', (e) => {
    const pill = e.target.closest('.agent-pill');
    if (pill) switchAgent(pill.dataset.agent);
});
```

## Open Questions

1. **Multi-turn context window size (N)**
   - What we know: The CONTEXT.md says "frontend sends last N messages as context"; N is not specified
   - What's unclear: Exact N value — too small loses coherence, too large hits token limits on Groq llama3-8b-8192 (8192 token context)
   - Recommendation: Default to last 10 turns (20 messages: 10 user + 10 assistant). This is well within Groq limits for typical chat messages and matches common chatbot patterns. Make it a named constant `MAX_CONTEXT_TURNS = 10` for easy tuning.

2. **History replay `replayMode` flag vs. a separate DOM-only helper**
   - What we know: `appendMessage` currently pushes to DOM and could push to history
   - What's unclear: Whether a `replayMode` boolean parameter is idiomatic enough or whether a separate `renderMessage(sender, text)` for DOM-only is cleaner
   - Recommendation: `replayMode = false` default parameter is the simplest change with the smallest diff; use it.
